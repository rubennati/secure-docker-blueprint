## Bug #1: Image tag `2.90` does not exist

**Symptom:** `docker compose up` fails with `not found` error.

**Root cause:** Docker Hub tag is `2.90-r3`, not `2.90`. Alpine packages use `<version>-r<revision>` format.

**Fix:** Changed `APP_TAG=2.90-r3` in `.env.example`.

**Lesson:** Always verify exact Docker Hub tags before committing. Use `curl` to list available tags.

## Bug #2: `failed to change group-id to dnsmasq: Operation not permitted`

**Symptom:** Container crashes with exit code 5.

**Root cause:** `cap_drop: ALL` removes SETUID and SETGID capabilities. dnsmasq needs these to drop privileges from root to the `dnsmasq` user on startup.

**Fix:** Added `SETUID`, `SETGID`, and `CHOWN` to `cap_add`.

**Lesson:** When using `cap_drop: ALL`, check if the process drops privileges on startup. Most services that start as root and then switch to a lower user need SETUID + SETGID.

## Bug #3: `LOUD WARNING: use --bind-dynamic rather than --bind-interfaces`

**Symptom:** Warning in logs about DNS amplification attack risk.

**Root cause:** `bind-interfaces` binds to specific IPs but doesn't prevent responses on other interfaces. `bind-dynamic` is the recommended replacement â€” it dynamically binds only to configured interfaces and prevents amplification.

**Fix:** Changed `bind-interfaces` to `bind-dynamic` in `dnsmasq.conf.tmpl`.

**Lesson:** dnsmasq security recommendations evolve. Check current best practices when deploying.

---

## Bug #4: `chown of PID file failed: please add capability CAP_CHOWN`

**Symptom:** Non-critical warning in logs about PID file ownership.

**Root cause:** `cap_drop: ALL` also removes CHOWN capability. dnsmasq tries to chown the PID file after dropping privileges.

**Fix:** Added `CHOWN` to `cap_add`.

## Bug #5: Healthcheck fails â€” `localhost is NXDOMAIN`

**Symptom:** Container goes unhealthy after `start_period`. Logs flooded with `query[A] localhost from 127.0.0.1` / `config localhost is NXDOMAIN` every 30 seconds (A + AAAA from healthcheck).

**Root cause:** The config uses `no-hosts` (intentionally â€” prevents container's `/etc/hosts` from leaking into DNS). But the healthcheck does `nslookup localhost 127.0.0.1`. Without `/etc/hosts`, dnsmasq doesn't know what `localhost` is â†’ NXDOMAIN â†’ healthcheck fails.

**Fix:** Added explicit localhost resolution to `dnsmasq.conf.tmpl`:

```
address=/localhost/127.0.0.1
address=/localhost/::1
```

**Lesson:** When using `no-hosts`, any name the healthcheck queries must be explicitly defined. A DNS server should always resolve `localhost`.
