# CrowdSec Bugfixes — 2026-04-15

## Bug #1: Traefik logs read but not parsed (RESOLVED)

**Symptom:** `cscli metrics` showed `Lines read: 1, Lines unparsed: 1`. No alerts or decisions generated from Traefik traffic.

**What worked:**
- CrowdSec Engine starts and registers with CAPI
- Collections installed (traefik, http-cve, appsec)
- Log file visible and readable inside container
- Manual `cscli explain --type traefik` parsed lines correctly
- Traefik access logs are valid JSON

**What didn't work (initially):**
- Live acquisition didn't parse incoming log lines
- Custom s00-raw parser to set `program=traefik` didn't help

**Root cause:** The `traefik-logs` parser v1.5 filtered on `evt.Parsed.program startsWith 'traefik'`. This field is populated by syslog/journald but NOT by file-based acquisition. Known issue: https://github.com/crowdsecurity/hub/issues/1677

**Attempts before fix:**
1. Added `program: traefik` to acquis.yaml labels — didn't set `evt.Parsed.program`
2. Created local s00-raw parser to set `program` + `message` — parser loaded but live parsing still failed
3. Removed custom parser, reverted to vanilla config

**Resolution:** Fixed by upstream hub update. After `cscli hub update && cscli hub upgrade`, the parser was updated and file-based acquisition works. Metrics now show 167 lines parsed, 0 unparsed. Real threats detected (http-probing, http-sensitive-files).

**Lesson:** When stuck on a known upstream issue, keep vanilla config and wait for the fix. Don't build workarounds that may break when the upstream fix lands.

## Revisit 2026-04-20 — diagnosis was likely wrong

A fresh install on new infrastructure with CrowdSec Engine v1.7.7 and parser `crowdsecurity/traefik-logs` v1.5 (the same version flagged as buggy above) produced acquisition metrics within ~5 minutes of first `docker compose up -d` — no hub update, no manual collection install, no restart. A clean-slate test (delete `./volumes/`, re-create, wait 21 minutes, run `cscli metrics show acquisition`) reproduced the same outcome: 5 lines read, 5 parsed, 9 bucket pours. Zero intervention.

What that means:

- The parser-filter hypothesis (`evt.Parsed.program startsWith 'traefik'` incompatible with file acquisition) does not hold under current engine + parser versions. File acquisition works out-of-the-box.
- The original "Fixed by upstream hub update" resolution was probably coincidental timing. CrowdSec runs `cscli hub update` + collection install asynchronously on first boot. During that window, `cscli metrics` shows only LAPI/Machines tables (no Acquisition/Parser/Bucket) because nothing has been parsed yet. The observer sees "broken", runs any command, waits a bit, re-checks — by then the async init has completed and metrics appear.
- The right diagnosis for "empty acquisition metrics right after first boot" is not "parser broken" but "first-boot init still in progress". Wait 5 minutes.

Action taken on the repo:
- `core/crowdsec/README.md` Verify section updated to state the 5-minute first-boot window explicitly and recommend `cscli metrics show acquisition` instead of grepping the full metrics output.
- This bugfix entry kept as-is above for the historical record, with this revisit appended so future readers see both sides.
