# Traefik bugs — 2026-04-20

## 1. Empty `integrations.yml` breaks the whole dynamic config load

**Symptom:**

Traefik logs show:

```
{"level":"error","providerName":"file",
 "error":"/etc/traefik/dynamic/integrations.yml: http cannot be a standalone element
          (type *dynamic.HTTPConfiguration)",
 "message":"Error while building configuration (for the first time)"}
```

Consequences:

- Dashboard returns 404 (`traefik.<domain>` → no router loaded)
- ACME wildcard router never loads → wildcard certificate is never requested
- Every app router that depends on file-provider middlewares fails

**Root cause:**

`ops/templates/dynamic/integrations.yml.tmpl` had this structure by default:

```yaml
http:
  middlewares:
    # --- CrowdSec ---
    # (all commented out)
    # --- Authentik ---
    # (all commented out)
```

When all middlewares below are commented out, the YAML parser reads
`http.middlewares` as `null`. Traefik's file provider refuses the file with
*"http cannot be a standalone element"*. Because the whole dynamic directory
is loaded as one unit, the error aborts the initial configuration build —
no routers, no middlewares, no certificates.

**Fix:**

Traefik v3 rejects both `http:` with null children AND `middlewares: {}`
with the same class of error ("cannot be a standalone element"). The file
is now **fully commented out by default** (same pattern as
`redirects.yml.tmpl`). The YAML document is effectively empty, which the
file provider accepts.

File layout: a single commented `http: / middlewares:` header at the top,
with both integration blocks (CrowdSec, Authentik) nested under it as
independent middleware definitions. To enable one: uncomment the header
plus the block. To enable both: uncomment the header plus both blocks.
This avoids the YAML duplicate-key pitfall of two separate `http:` roots.
Instructions at the top of the file.

**Apply to a running install:**

```bash
cd core/traefik
bash ops/scripts/render.sh
docker compose up -d --force-recreate traefik
docker compose logs traefik 2>&1 | grep -i error
# should be empty now
```

## 2. Wildcard vs. per-domain certificate strategy was under-documented

**Symptom:**

After a fresh install with `ACME_WILDCARD_DOMAIN` set, the wildcard
certificate was never issued. App routers had no working TLS.

**Root cause:**

Two paths exist — both are valid setups — but the setup docs only
mentioned the variable existence, not the implications:

- **Wildcard mode:** `ACME_WILDCARD_DOMAIN` + `CF_DNS_API_TOKEN` set →
  Traefik issues one cert for `*.domain`. Apps leave the
  `tls.certresolver` label **commented out** and rely on the wildcard
  via SNI.
- **Per-domain mode:** `ACME_WILDCARD_DOMAIN` empty → apps must
  **uncomment** the `tls.certresolver` label to request their own cert.

Without knowing which mode is active, the per-app compose labels end up
in the wrong state and no cert gets issued.

**Fix:**

- `core/traefik/README.md` — new section "Certificate strategy —
  wildcard vs. per-domain" with the concrete steps per path plus
  verify commands.
- `core/traefik/.env.example` — ACME block restructured to state the
  two paths explicitly and which env vars go with each.

**Verify after setup:**

```bash
# Did Traefik receive any certs?
docker compose exec traefik cat /etc/traefik/acme/acme.json \
  | jq '.[] | .Certificates[]?.domain // "no certs yet"'

# Dashboard reachable via HTTPS?
curl -I https://<TRAEFIK_DASHBOARD_HOST>

# Current ACME/cert state in logs:
docker compose logs traefik 2>&1 | grep -iE "acme|cert|challenge" | tail -20
```

Note: bug #1 was masking bug #2. Fix #1 first, then verify #2 against the
strategy you picked.
