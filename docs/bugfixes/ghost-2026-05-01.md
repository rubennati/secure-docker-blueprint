# Ghost — ERR_INVALID_ARG_TYPE at MySQL auth

**App:** `apps/ghost/`  
**Ghost version:** 6.27.0-alpine  
**MySQL version:** 8.4

## Symptom

Ghost entered a crash loop. Two distinct failure phases visible in the logs:

**Phase 1 (first restarts):**
```
ECONNREFUSED 172.x.x.x:3306
```
MySQL not yet ready when Ghost started. This resolved itself once MySQL passed its healthcheck, but Ghost kept crashing.

**Phase 2 (subsequent restarts):**
```
TypeError [ERR_INVALID_ARG_TYPE]: The "data" argument must be of type string
or an instance of Buffer, TypedArray, or DataView.
Received an instance of Object
    at Object.sha1 (node_modules/mysql2/lib/auth_41.js:31:8)
```

## Root Cause

The compose file used `database__connection__password__file` as an environment variable.

Ghost uses nconf's double-underscore notation for hierarchical config. Each `__` segment creates a new nesting level. So:

```
database__connection__password__file = /run/secrets/DB_PWD
```

resolves to:

```json
{ "database": { "connection": { "password": { "file": "/run/secrets/DB_PWD" } } } }
```

The password is an **object** `{file: '/run/secrets/DB_PWD'}` — not the string content of the file. mysql2 received this object at the sha1 authentication step and crashed.

Ghost does **not** support `_FILE` env var semantics natively (unlike MySQL/Postgres official images). The same issue applied to `mail__options__auth__pass__file`.

## Fix

Added `ops/entrypoint.sh` — a wrapper that reads the Docker Secrets at container startup, exports them as the plain env vars Ghost expects, then hands off to the original Ghost entrypoint:

```sh
#!/bin/sh
set -e

if [ -f /run/secrets/DB_PWD ]; then
  export database__connection__password
  database__connection__password="$(cat /run/secrets/DB_PWD)"
fi

if [ -f /run/secrets/GHOST_MAIL_PWD ]; then
  export mail__options__auth__pass
  mail__options__auth__pass="$(cat /run/secrets/GHOST_MAIL_PWD)"
fi

exec docker-entrypoint.sh node current/index.js
```

In `docker-compose.yml`:
- `entrypoint: ["/bin/sh", "/entrypoint.sh"]` added to app service
- `database__connection__password__file` and `mail__options__auth__pass__file` removed from environment
- `./ops/entrypoint.sh:/entrypoint.sh:ro` added to volumes

This follows the blueprint's standard pattern for images that don't support `_FILE` — see `docs/standards/security-baseline.md`.

# Ghost — `activitypub-migrate` connection refused on startup

**App:** `apps/ghost/` (ActivityPub overlay)  
**Date:** 2026-05-02

## Symptom

```
error: dial tcp 172.25.0.2:3306: connect: connection refused
```

`activitypub-migrate` failed with exit 1 despite `depends_on: db: condition: service_healthy` in the compose file. `ghost-db` showed `Healthy` at roughly the same time.

## Root Cause

`mysqladmin ping -h localhost` on Linux resolves to a **Unix domain socket**, not TCP. MySQL's Unix socket becomes available slightly before the TCP port `3306` is bound. The healthcheck passed (socket ready) while TCP was still not accepting connections. The `activitypub-migrate` container started, tried to connect via TCP, and got `connection refused`.

## Fix

Changed the healthcheck from `-h localhost` to `-h 127.0.0.1`. An IP address always forces TCP — the check now passes only when the TCP port is actually ready.

```yaml
healthcheck:
  test: ["CMD-SHELL", "mysqladmin ping -p\"$(cat /run/secrets/DB_ROOT_PWD)\" -h 127.0.0.1"]
```

The password flag is required because the blueprint uses `MYSQL_ROOT_PASSWORD_FILE` (Docker Secrets) instead of the plain `MYSQL_ROOT_PASSWORD` env var. The MySQL entrypoint resolves `*_FILE` variables only during container initialisation, not for subsequent healthcheck runs.

# Ghost — `ERR_TOO_MANY_REDIRECTS` on ActivityPub endpoints

**App:** `apps/ghost/` (ActivityPub overlay)  
**Date:** 2026-05-02

## Symptom

Authenticated requests to `/.ghost/activitypub/v1/*` triggered an infinite redirect loop in the browser. Unauthenticated `curl` requests returned clean HTTP 403 (no loop), masking the problem in initial testing.

## Root Cause

ActivityPub uses a `behindProxy` middleware (`src/lib/behindProxy.ts`) that reconstructs the absolute request URL from the `X-Forwarded-Proto` header. When the header is missing, `proto` falls back to `http`, causing a redirect loop:

1. ActivityPub reconstructs URL as `http://` → issues redirect
2. Traefik HTTP→HTTPS entrypoint picks it up → redirects back
3. Loop until browser aborts

A secondary factor: ActivityPub's trailing-slash middleware fires **before** the request logger (`src/app.ts` line 699 vs. 709). Redirected requests never reach the logger, so nothing appears in the ActivityPub container logs.

## Fix

Added an explicit `customrequestheaders` middleware to the ActivityPub Traefik router. The official ghost-docker setup uses Caddy as the internal webserver, which forwards `X-Forwarded-Proto: https` automatically. Traefik requires the header to be set explicitly.

```yaml
labels:
  - "traefik.http.middlewares.${COMPOSE_PROJECT_NAME}-ap-proto.headers.customrequestheaders.X-Forwarded-Proto=https"
  - "traefik.http.routers.${COMPOSE_PROJECT_NAME}-ap.middlewares=...,${COMPOSE_PROJECT_NAME}-ap-proto"
  - "traefik.http.services.${COMPOSE_PROJECT_NAME}-ap.loadbalancer.passhostheader=true"
```

`passhostheader=true` ensures the original `Host` header reaches ActivityPub unchanged, required for correct URL reconstruction.

## Browser cache note

Browsers can retain redirect chains across sessions. After applying this fix, a regular browser window that previously hit the loop may continue looping due to cached redirects. **Always verify in a fresh private/incognito window** — if it works there, clearing browser cache or using a different browser resolves it for existing sessions.

# Ghost — Admin login blocked (SMTP TLS mismatch)

**App:** `apps/ghost/`  
**Date:** 2026-05-02

## Symptom

Ghost admin login page appeared, but after entering credentials the verification email never arrived. SMTP logs showed:

```
48025FB4757F0000:error:0A00010B:SSL routines:tls_validate_record_header:wrong version number
```

## Root Cause

Ghost 6 sends an email verification code for every new-device login. Without a working SMTP relay, admin access is blocked after the initial session cookie expires.

The error `tls_validate_record_header:wrong version number` means a TLS/port mismatch: the client attempted an SSL/TLS handshake on a port that expects STARTTLS (or vice versa).

`mail__options__secure` was hardcoded to `"true"` in `docker-compose.yml`, which enables SSL/TLS and expects port 465. The `.env.example` defaulted to port `465`. The actual relay (Brevo) uses port 587 with STARTTLS, which requires `secure: false`.

| Mode | Port | `secure` value |
|------|------|----------------|
| SSL/TLS | 465 | `true` |
| STARTTLS | 587 | `false` |

## Fix

Made `mail__options__secure` configurable via the `GHOST_MAIL_SECURE` env var instead of hardcoding it in `docker-compose.yml`. Updated `.env.example` defaults to Brevo/STARTTLS:

```
GHOST_MAIL_HOST=smtp-relay.brevo.com
GHOST_MAIL_PORT=587
GHOST_MAIL_SECURE=false
GHOST_MAIL_USER=<brevo-account-email>
# .secrets/mail_pwd.txt = SMTP key from Brevo → Transactional → SMTP & API
```
